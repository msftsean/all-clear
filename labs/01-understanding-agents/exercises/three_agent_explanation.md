# Three-Agent Architecture Explanation

All Clear uses a three-stage pipeline because incident triage needs both language understanding and auditable safety controls. A monolithic agent could be asked to classify, route, assign severity, create records, search knowledge, and write responses in one prompt. That is hard to test and risky during a surge. All Clear instead separates the work into QueryAgent, RouterExecutor, and ActionAgent, with bounded authority enforced by code structure.

The first stage is the QueryAgent. It receives one inbound signal, such as "Power line down across Main St, sparking near a school" or "When will power be restored on Elm St?" Its job is to produce a structured `SignalClassification`: intent, `SignalCategory`, target queue suggestion, confidence, entities, severity indicators, PII flags, sentiment, and urgency indicators. It classifies only. It does not open an `AC-####` incident, attach a report, search knowledge, assign `SEV1`-`SEV4`, or decide whether to escalate.

The second stage is the RouterExecutor. It is intentionally not an LLM agent. It is deterministic workflow code with zero LLM calls. It takes the classification and applies auditable policy: dedup against open incidents in the same `intent_category`, choose `ATTACH_TO_INCIDENT` at or above the 0.83 cosine threshold or `OPEN_INCIDENT` below it, map severity and SLA, and apply escalation rules. This keeps severity away from model vibes. A signal involving gas, fire, trapped people, total outage, or a statutory clock cannot be talked down by a caller or by prompt drift.

The third stage is the ActionAgent. It acts only through three tools: `create_incident`, `search_knowledge`, and `generate_sitrep`. On the `OPEN_INCIDENT` path, it can create a new incident, retrieve runbook or SOP knowledge, and generate a citation-grounded sitrep. On the `ATTACH_TO_INCIDENT` path, the signal becomes a report on an existing incident, magnitude increments, and the system sends a short acknowledgment without knowledge search. This keeps outage surge handling fast and predictable.

The orchestrating pipeline preserves the complete flow from signal to incident action. A signal is the inbound communication. An incident is the real-world event. A report is the association of the signal to an incident. The incident has a queue, severity, SLA clock, magnitude, and status. Queue names are `field-operations`, `customer-comms`, `compliance-desk`, `engineering`, and `escalations`.

This pattern scales better than a monolithic agent because each component has a narrow contract. Classification quality, routing quality, and action quality can be measured separately. It also improves security: QueryAgent cannot mutate records, RouterExecutor cannot be prompt-injected because it has no prompt, and ActionAgent cannot bypass escalation or edit arbitrary data. The result is clearer system behavior, better surge response, and more predictable outcomes on the ClearBoard.
