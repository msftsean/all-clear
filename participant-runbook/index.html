What you're going to build

All Clear is an incident-triage agent: a signal comes in — someone reports a problem in plain language — and an agent turns that speech into a structured, auditable incident, then drafts a public update that waits for a human's approval. Over Day 2 you build that agent yourself, end to end.

Six stages, one repo, one deployed URL. You start from a spec'd scaffold and finish with a deployed, evaluated, gateway-fronted service. Each stage builds on the last — you don't get handed the finished product, you ship it.

the six domains of a production agent

Across these stages you put all six in place: identity, observability, evals, safety, memory & state, and cost governance — the same six domains from the Day 1 talk.

how you work

In a pre-configured GitHub Codespace with Azure wired up. The detailed instructions for each stage live in the repo under labs/ — this workbook is your map and checklist.

No finished app to peek at

That's deliberate. This is a build lab, not a demo — the whole point is that you make it work. Use the API docs / health buttons above to test your own running service as you go.

📖 Lab 00 — Environment Setup  ·  📖 Lab 03 — Spec-Driven Development

S1 · A working agent on Agent Framework 90 min

Goal: a working agent with the core building blocks in place — model, tools, orchestration, plus the first pass at memory, safety, and evaluation.

  1. Understand the multi-agent shape: QueryAgent → RouterExecutor → ActionAgent — classify, route (deterministic, no AI), then act.
  2. Wire up the three-agent pipeline with handoffs and session context so a multi-turn conversation holds together.
  3. Make a signal flow through: classification (model), routing (rules), action (model) — with the human-approval boundary intact.
  4. Make the model swappable. Put your chat model behind a small interface (an ordered list, primary first) instead of hard-coding one deployment, and expose GET /api/health/models so an operator can see which model is serving. The reference build does this with a FailoverChatClient that advances to a standby model on a model-unavailability error (404/401/403/503).
Why model-agnostic matters now

A frontier model can be restricted or pulled overnight. For a regulated triage system that's operational risk, not just inconvenience — so design for it: route around an unavailable model, but never route around a 429 rate limit (retry instead) or a content-safety block (that's the system working). The goal is continuity without a safety bypass.

Check yourself

A plain-language report becomes a classified, routed, structured incident, and anything public-facing stops at an approval gate.

Why the router uses no AI

Severity, SLA, and escalation are safety-critical, so they're computed by deterministic rules — not the model. Nobody can talk the system into a lower severity. You'll harden this further in S3.

S2 · Agentic retrieval 60 min

Goal: multi-source agentic retrieval over articles, policies, and records — grounded answers with citations.

  1. Stand up hybrid (keyword + vector) search over the knowledge base.
  2. Build a RetrieveAgent that pulls the right sources and returns citations, not just prose.
  3. Fold retrieval into the pipeline so the action step can cite what it used.
Check yourself

A query returns relevant KB articles and the response carries citation pills ([AC-####]) you can trace.

📖 Lab 04 — Build the RAG Pipeline

S3 · MCP server + red-team 75 min

Goal: expose your tools through a discoverable MCP server, then run a red-team pass to prove it's safe.

  1. Configure the Azure MCP integration and understand the tool/resource model.
  2. Expose All Clear as an MCP server (e.g. create_incident, search_knowledge, generate_sitrep) and call it from Copilot Agent Mode.
  3. Red-team it: attempt PII leakage, prompt injection, and a "talk the router down" attack — confirm PII is redacted, jailbreaks are rejected, and severity can't be influenced.
Check yourself

@all-clear tools respond in VS Code, and your three red-team probes all fail to break the system.

📖 Lab 07 — MCP Server  ·  📖 Lab 02 — Azure MCP Setup

S4 · APIM as the AI gateway 60 min

Goal: put a production AI gateway in front of your agent — rate limits, token budgets, JWT auth, and full observability.

  1. Provision an APIM Consumption gateway (serverless, pay-per-call) and import your backend's OpenAPI.
  2. Throttle callers with rate-limit-by-key and cap LLM spend with azure-openai-token-limit.
  3. Require a validate-jwt token, then wire APIM diagnostics to Application Insights.
Check yourself

A signal triages through the gateway; the 21st call returns 429; calls with no token return 401; traffic shows up in App Insights.

Why Consumption SKU

No fixed cost, provisions in minutes, and it's verified to handle a whole class creating gateways at once. Use the -by-key policy variants; semantic caching is out of scope on this tier.

📖 Lab 08 — APIM as the AI Gateway

S5 · Deploy + eval gating CI 60 min

Goal: deploy on Azure Container Apps with the eval suite gating CI — one deployed URL you own.

  1. Containerize the pipeline and deploy with azd up (infra + app in one command).
  2. Confirm the deployment is healthy, then point the API docs / health buttons at the top of this page to your URL.
  3. Wire the eval suite into CI so a regression blocks the merge — that's the gate that makes it production-grade.
Check yourself

Your agent answers at a public URL, the health check is green, and CI runs the evals and fails on regression.

A2 · Phone call-in optional

The vision: dial a real phone number, describe your problem out loud, and All Clear answers and speaks back — running the exact same triage you've been doing by hand. The plumbing for this is already built in the code (Azure Communication Services + a media bridge to the realtime model); it just isn't switched on in this environment yet.

  1. You call a phone number.
  2. The call audio is streamed to the same realtime model used for in-browser voice.
  3. It transcribes you, runs the triage pipeline, and speaks the response back — full duplex, just like a real dispatcher.
Available to you right now?

Not yet enabled. Like voice, it needs the realtime model and a phone number configured. Nothing for you to do in this workbook — just know it's on the roadmap.
